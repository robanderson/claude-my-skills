# D-0008 — Audit: do the tournament runners share the D-0005 "nonzero exit discards a completed result" pattern?

- id: D-0008
- status: wontfix          # audited; the runners do NOT have the D-0005 bug, and the engine's exit=0 gate is intentional for the agentic path
- severity: sev3           # precautionary audit prompted by D-0005, not a discovered defect
- area: runner
- discovered: 2026-06-15
- reporter: rob
- provenance (NON-DURABLE): follow-up audit requested when D-0005 (fl-bench) was fixed — "audit the
  tournament runners (glm/minimax/codex/local) for the same nonzero-exit-discards-completed-result pattern".

## Problem (the hypothesis audited)

D-0005 was: `fl-bench` scored a **completed** claude-family generation as FAIL whenever the `claude`
CLI exited nonzero (root cause: a 256 output cap below the 1024 thinking-budget floor → API 400; and a
glm-5.1 completed result discarded on exit code). The question: do the four tournament runners
(`bin/glm-run.sh`, `minimax-run.sh`, `codex-run.sh`, `local-run.sh`) — and the engine path that
consumes them — drop a good attempt the same way?

## Findings (audit, not a fix)

**The runners do NOT replicate the bug, for two independent reasons:**

1. **No sub-1024 output cap is set.** None of the four runners export `CLAUDE_CODE_MAX_OUTPUT_TOKENS`
   (they pass `--max-turns` for the agentic loop, not an output-token cap). So D-0005's *actual trigger*
   — a cap below the extended-thinking budget floor producing a 400 — cannot occur here.

2. **The engine's `exit=0` requirement is an intentional, load-bearing health gate for the AGENTIC
   path, unlike fl-bench's pure-generation path.** Each runner faithfully records
   `FARNSWORTH-<PROV>-DONE exit=$RC` and does not itself discard anything; the engine's
   `provCheckShell` (in `workflows/tournament.mjs`) requires `DONE exit=0` (plus the PROVENANCE marker
   and no TIMEOUT/ERROR) for `P=1`, **and separately** requires a non-empty deliverable (`D>0`).

   The two contexts differ in a way that makes the gate correct here but wrong there:
   - **fl-bench (D-0005):** a *pure generation* (`--allowedTools ''`, no `--max-turns`). The "result"
     is just a token count; a nonzero CLI exit on a completed result is a pure CLI quirk, so salvaging
     it is unambiguously right.
   - **Tournament runners:** an *agentic loop* (`--max-turns`, `--allowedTools "Bash Read Write Edit"`)
     that writes files. A nonzero exit there genuinely signals an incomplete run — a wall-clock timeout
     (`124`, also marked `…-TIMEOUT`), max-turns exhaustion mid-task, or a crash mid-write — which can
     leave a **partial/garbage deliverable**. The `D>0` check only proves a file *exists*, not that the
     agent *finished*; the `exit=0` gate is what distinguishes "completed cleanly" from "died mid-write".
     Relaxing it (accepting `D>0` regardless of exit) would admit broken deliverables into the
     ranking — a sev1 outcome-corruption risk, the exact failure class the dogfood backlog guards.

   `codex-run.sh` already demonstrates the *correct* targeted version of the D-0005 idea: it force-fails
   on a model/auth/version error **only when `$LAST` is empty** (no deliverable), so a successful run is
   never force-failed. That is "tie the exit signal to deliverable presence", applied where it is safe.

## Repro / verification

- `grep -L CLAUDE_CODE_MAX_OUTPUT_TOKENS bin/*-run.sh` → all four listed (none set the cap).
- Read `workflows/tournament.mjs` `provCheckShell` / `stageAndValidate`: `P` needs `DONE exit=0`; `D>0`
  is a *separate* gate; pool inclusion is `[ "$D" -gt 0 ] && [ "$P" -eq 1 ]`.

## Suspected fix

None. Closing **wontfix**: the runners do not have the D-0005 bug, and the engine's `exit=0` gate is the
correct conservative health check for an agentic attempt (a timed-out/crashed agent's half-written file
must not win a tournament). If a *specific* "completed cleanly but exited nonzero with a valid
deliverable" case is ever observed in a real run, re-open with that evidence and consider the
codex-style targeted relaxation (accept `D>0` only when a clean-completion signal is present).

## Resolution

Closed **wontfix** 2026-06-15 — audit only, no code change. Documented here so the runners are not
re-investigated for this pattern. Related: [[D-0005]].
