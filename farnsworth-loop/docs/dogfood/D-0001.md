## Problem

In the tournament that designed the verdict-persistence feature, every candidate had to choose how
`tournament.mjs` (which runs inside the **dynamic-workflow sandbox**) should write files. Eight of
nine candidates used `import { writeFileSync } from 'node:fs'`. One candidate (blind **B**) instead
wrote files via a cheap haiku agent running a Bash heredoc — the engine's existing
`buildContext`/`stageAndValidate` pattern.

The blind Opus reviewer **ranked B last** and called its mechanism "wrong", on the stated premise
that "`node:fs` is plainly available." It is **not**: a later environment probe proved the workflow
sandbox has no `node:fs`, no `import()`, no `process`. So the judge confidently inverted the
ranking — it rewarded the *familiar-looking* API over the *only mechanism that actually works*. The
shipped feature (PR #2) uses exactly B's approach. A judge that selects a non-functional winner on
an unchecked environment assumption is a sev1 against the whole point of the loop: trustworthy
ranking.

## Durable evidence (verbatim excerpt — survives `.runs/` deletion)

> Round-1 reviewer, on blind candidate B:
> "B is last and the only one with a genuine constraint VIOLATION: it routes all file writes through
>  a haiku AGENT shell call on the false premise that fs is unavailable — directly breaking 'no new
>  agents / negligible cost' for a fix where **node:fs is plainly available**. Good honesty about
>  its own risks, but **the core mechanism is wrong for this task**."

Counter-fact established the same session by direct probe of the workflow sandbox:

> `await import('node:fs')` → **"Error: import() is not available in workflow scripts."**
> `typeof process === 'undefined'`, `typeof require === 'undefined'`.

i.e. the judge's load-bearing sentence ("node:fs is plainly available") is false, and the candidate
it ranked last was the only correct one.

## Repro

1. Run any `@@FL` whose correct solution depends on an environment constraint the model can't see
   from the code alone (here: "workflow scripts have no Node API").
2. Include one candidate that honours the real constraint via an unfamiliar-but-correct mechanism
   and others that use the familiar API.
3. Observe the blind reviewer reward the familiar API and rank the correct candidate low, justifying
   it with a generic "idiomatic / available" rationale.

## Suspected fix (for the worker who claims this)

Add a line to the review rubric (`skills/farnsworth-loop/references/review-rubric.md` + the Phase 3
prompt): the judge must **not assume an execution environment it cannot see**, must score against
the task's *stated* runtime constraints, and should treat "relies on a capability the task did not
establish is available" as a risk, not a strength. Consider having the brief state known sandbox
constraints explicitly when the task is workflow/engine code.

## Resolution

Fixed in this PR. Added an unconditional, model-blind scoring rule in two durable places: a new
item 5 in the **Shared scoring method** of `skills/farnsworth-loop/references/review-rubric.md`, and
one sentence in the `judgePrompt(...)` template in `workflows/tournament.mjs`. The rule tells the
blind judge to score against the task's *stated* runtime, to treat reliance on an unestablished
capability as a risk, and to treat an unfamiliar but constraint-honouring mechanism as correct
unless it can name a concrete failure — so it no longer rewards a familiar-looking `node:fs` plan
over the correct haiku+Bash mechanism, nor penalises the correct one. The rubric also records the
verified sandbox fact (dynamic-workflow scripts have no `node:fs`/`require`/`import()`/`process`) as
a general property of this engine, stated without revealing any candidate's model — blind review is
preserved.
