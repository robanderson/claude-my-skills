# D-0007 — Task text after a leading `@@FL` marker is silently dropped (`task:""`)

- id: D-0007
- status: open
- severity: sev2            # the whole task can be silently lost -> a naive run dispatches attempts with an EMPTY task
- area: parse
- discovered: 2026-06-15
- reporter: rob
- provenance (NON-DURABLE): interactive `bin/fl-parse.mjs` probes 2026-06-15, and live this session —
  `@@FL two pass, 2 opus, …, Please review the review at the end of round 1 …?` returned `task:""`
  (the entire "Please review …" task was dropped; the SKILL operator had to supply it by hand).

## Problem

`fl-parse.mjs` derives the task as **the text BEFORE the marker** (then strips the spec / Top-Mixed /
sigil from it). When the user writes the marker **first** — `@@FL <spec>, <task>`, the habitual form
in this project — there is no pre-marker text, so after the spec is stripped the task is **empty**,
with **no error**. A naive caller would then dispatch every attempt with an empty task (a wasted,
meaningless tournament). Today this is only masked because the SKILL operator notices `task:""` and
hand-builds the task for the workflow — but the parser's contract is silently broken for the most
common invocation shape.

## Durable evidence (verbatim excerpt — survives `.runs/` deletion)

> `node bin/fl-parse.mjs '@@FL 2 opus, fix the parser bug'`        → `task=''`  (task LOST)
> `node bin/fl-parse.mjs '@@FL fix the parser bug'`                → `task='' , needsGate=true`
> `node bin/fl-parse.mjs 'fix the parser bug @@FL 2 opus'`         → `task='fix the parser bug'`  ✓
> `node bin/fl-parse.mjs 'fix the parser bug @@FL:2'`              → `task='fix the parser bug'`  ✓
>
> i.e. the task survives ONLY when it precedes the marker; anything after a leading `@@FL` is dropped.

## Repro

1. `node bin/fl-parse.mjs '@@FL 2 opus, do the thing'` → observe `"task": ""` (the "do the thing" is gone).
2. Compare `node bin/fl-parse.mjs 'do the thing @@FL 2 opus'` → `"task": "do the thing"`.

## Suspected fix (for the worker who claims this)

Capture the task from **both sides** of the marker: after removing the spec, Top-Mixed keyword,
prose mode phrase ([[D-0006]]), `<n> grand loop[s]` phrase ([[D-0003]]) and the sigil itself, treat
the remaining text on EITHER side of the marker as the task (concatenate the non-empty remainders, or
take the non-empty side). Be careful not to swallow the model spec or re-introduce a stripped
directive. Add unit cases to `bin/fl-parse.test.mjs` for a leading-marker task (`@@FL <spec>, <task>`
and `@@FL <task>`), confirming the task is captured and the spec is still parsed. Note the strong
interaction with D-0003/D-0006: all prose directives must be stripped from the post-marker remainder
too, or they would leak into the captured task.
