# D-0006 — Prose `two pass` / `two-pass` ignored; silently runs single pass

- id: D-0006
- status: done
- severity: sev2            # wrong MODE, no error: a two-pass request silently runs as single pass (skips the guided round + final rank)
- area: parse
- discovered: 2026-06-15
- reporter: rob
- provenance (NON-DURABLE): interactive `bin/fl-parse.mjs` probes 2026-06-15, and live this session —
  the invocation `@@FL two pass, 2 opus, 2 sonnet, 2 codex high, 2 glm 5.2, 2 minimax, z=1 1 grand loop, …`
  parsed to `mode:1` despite the leading words "two pass".

## Problem

`fl-parse.mjs` only sets the pass count `M` from the **numeric sigil** (`@@FL:N:2`) or the prose
marker (`farnsworth loop:N:2`). The bare prose words **`two pass`** / **`two-pass`** (and, by
symmetry, `single pass` / `one pass`) are **not recognised** — they fall through, `mode` stays at the
default `1`, and **no error is raised**. A user who writes "two pass" therefore silently gets a
single-pass run: the entire second guided round, the round-2 guidance, and the final rank are
skipped. Because the prose model-spec (`2 opus, …`) IS recognised, the run looks like it understood
the message, masking the dropped mode.

## Durable evidence (verbatim excerpt — survives `.runs/` deletion)

> `node bin/fl-parse.mjs '@@FL two pass, 2 opus, 2 sonnet do X'` → `mode=1` (expected 2)
> `node bin/fl-parse.mjs '@@FL two-pass, 2 opus, 2 sonnet do X'` → `mode=1` (expected 2)
> `node bin/fl-parse.mjs '@@FL:4:2 2 opus, 2 sonnet do X'`       → `mode=2`  ✓ (sigil works)
> `node bin/fl-parse.mjs 'do X farnsworth loop:4:2'`            → `mode=2`  ✓ (prose marker works)
>
> So the numeric `:M` forms set the mode; the plain English `two pass` does not — it is dropped with
> no error, and the run proceeds single-pass.

## Repro

1. `node bin/fl-parse.mjs '@@FL two pass, 2 opus, 2 sonnet — do X'` → observe `"mode": 1`.
2. Compare `node bin/fl-parse.mjs '@@FL:4:2 …'` → `"mode": 2`.

## Suspected fix (for the worker who claims this)

In `bin/fl-parse.mjs`, recognise a bare prose mode phrase: `two[\s-]?pass` → `M=2`,
`single[\s-]?pass` / `one[\s-]?pass` → `M=1`. Strip the phrase from the task/spec scan (same pattern
as the `<n> grand loop[s]` / Top-Mixed strips). Reconcile prose vs the numeric sigil exactly like the
Z reconciliation added in D-0003: the **sigil `:M` wins**, and a sigil-vs-prose **disagreement is a
loud error** (never silently guessed). Add unit cases to `bin/fl-parse.test.mjs` for `two pass`,
`two-pass`, `single pass`, sigil-wins, and the conflict. (Pairs naturally with [[D-0007]], which
drops the task text after a leading marker.)

## Resolution

**Fixed** (branch `rob/dogfood-fix-d0006-d0007`, with [[D-0007]]). `fl-parse.mjs` now recognises a prose
pass directive (`two pass`/`two-pass` → M=2; `single pass`/`one pass` → M=1) and reconciles it with the
sigil `:M` exactly like the Z/grand-loop logic: **the sigil wins; a sigil-vs-prose disagreement is a loud
error; the prose phrase alone sets the mode.**

**Critically anchored** (an adversarial-review catch): the directive is matched ONLY when it sits
**immediately adjacent to the marker** (right after it, then a clause boundary — comma/semicolon, the
spec's leading digit, or end-of-side; or right before it). The first cut scanned the whole message and
flipped the mode on — or raised a false pass-count conflict that *refused the run* over — ordinary task
vocabulary like `two-pass compiler`, `single-pass renderer`, `one pass over the data`, `the two-pass
build`. The anchored form (`PASS_AFTER_RX` / `PASS_BEFORE_RX`) leaves such adjectives untouched while
still catching `@@FL two pass, …` and `… @@FL:4 single pass`. The matched directive is stripped from its
own side of the marker (never via a whole-message strip that would eat a task's `two-pass`).

Tests: `bin/fl-parse.test.mjs` extended to **293 cases (was 245), all green** — covering the directive
forms, the sigil-wins/conflict matrix, and over-match regressions (`build a two-pass compiler @@FL:4` →
n=4/mode=1/task kept; `replace the two-pass build … @@FL:5:1` → no false conflict; `single-pass renderer`
kept; mid-task `two-pass` after a spec kept). Verified live.
