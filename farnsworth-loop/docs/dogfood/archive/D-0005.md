# D-0005 — fl-bench scores a *completed* claude-family call as FAIL on nonzero CLI exit (haiku, glm-5.1)

- id: D-0005
- status: done
- severity: sev2            # degraded but usable: the sweep still runs, but callable, working models are mis-reported as FAIL and a real, valid measurement is discarded
- area: infra
- discovered: 2026-06-15
- reporter: rob
- provenance (NON-DURABLE, may be gc'd): `node bin/fl-bench.mjs --models all` on 2026-06-15
  (records persisted at `2026-06-15T14:48:04+12:00`; transient logs `.bench/run.log` +
  task output `b71t0cntu.output`). `.bench/` is gitignored, so the offending lines are pasted verbatim below.

## Problem

`runClaudeFamily()` in `bin/fl-bench.mjs` treats **any** nonzero exit of the nested `claude -p`
as a hard failure and throws away stdout:

```js
// bin/fl-bench.mjs ~L345-348
if (r.status !== 0) {
  const tail = ((r.stderr || '') + (r.stdout || '')).trim().slice(-300)
  return { ok: false, secs, tokens: 0, estimated: false, error: `claude exit ${r.status}: ${tail || 'no output'}` }
}
```

But for both `anthropic:haiku` and `glm:glm-5.1` the nested `claude` actually printed a **completed**
result JSON to stdout (`"terminal_reason":"completed"`) and still exited `1`. The exit-code branch
fires *before* `parseClaudeJsonTokens(r.stdout)` is ever called, so the usage data sitting in stdout
is discarded and the model is recorded `FAIL`. Two distinct signals share this one symptom:

1. **glm-5.1 — a valid measurement is destroyed.** The result tail carries `"outputTokens":256` over a
   real `206.71s` cold window (≈1.24 t/s) and `terminal_reason:"completed"` — a complete, usable
   generation. The bench discarded it purely because the CLI exit code was `1`. (Contrast: glm-5.2 /
   glm-4.7 / glm-4.5-air all passed, so this is glm-5.1-specific, not an auth/key problem.)
2. **haiku — empty model turn + nonzero exit.** The tail shows `"modelUsage":{}` and `"iterations":[]`
   (no assistant turn) in ~`1.74s`, exit `1`. Here there is genuinely nothing to salvage (0 tokens), but
   the *same* nonzero-exit-first handling masks the real cause and reports an opaque "claude exit 1"
   instead of "claude completed but produced no output / empty modelUsage". (opus + sonnet passed, so
   it is haiku-specific — likely the `--model haiku` alias × `CLAUDE_CODE_MAX_OUTPUT_TOKENS=256` soft
   cap × empty `--allowedTools ''` interaction, to be confirmed.)

Net effect: the all-models sweep reported 21/31 OK and listed haiku + glm-5.1 among the failures, when
glm-5.1 in fact produced a fully measurable result and haiku failed for a different, hidden reason.

## Suspected fix (for the worker who claims this)

- **Salvage completed results (primary, smallest).** In `runClaudeFamily`, when `r.status !== 0`,
  do **not** bail immediately. First run `parseClaudeJsonTokens(r.stdout)`; if it yields a parsed,
  completed result with `output_tokens > 0`, return `{ ok: true, tokens, ... }` (the generation
  happened; the CLI exit code is unreliable). Only hard-fail when stdout has no usable completed
  result. Preserve the timeout (`status === 124`) and the "parsed but 0 tokens" failure paths.
- **Diagnose the nonzero exit (secondary).** Capture one full `claude -p --model glm-5.1 ...` and
  `--model haiku ...` stdout+exit to find *why* a completed result exits `1` (candidates: the
  `CLAUDE_CODE_MAX_OUTPUT_TOKENS=256` soft cap tripping a nonzero "stopped early" exit; an
  alias/availability quirk for haiku producing an empty `modelUsage:{}`). Surface the real reason in
  the recorded `error` (e.g. "completed but empty modelUsage") instead of an opaque "claude exit 1".
- **Audit the tournament runners for the same pattern.** `bin/glm-run.sh` / `minimax-run.sh` (and the
  codex/local runners) may likewise treat a nonzero `claude`/CLI exit as failure even when a completed
  result was produced — worth a quick check while here (not claimed by this item; file separately if real).
- **Regression:** a claude-family call whose stdout contains a completed result with `usage.output_tokens > 0`
  must be scored `OK` (tokens salvaged) even when the CLI exit code is nonzero; an empty-`modelUsage`
  completion must report a descriptive error, not a bare exit code.

## Resolution

**Fixed** (PR #9, branch `rob/dogfood-D-0005`). Implemented directly (no `@@FL` sigil was given) with a
live root-cause repro first, an isolated git **worktree** (a parallel session held the shared tree), and
an **adversarial multi-dimension review workflow** that caught a real regression in the first cut (below).

**Root cause (confirmed by live repro), two distinct cases:**
- **haiku** was not a "completed-but-nonzero-exit" case at all — the old `CLAUDE_CODE_MAX_OUTPUT_TOKENS=256`
  cap is **below the extended-thinking budget floor of 1024**, so the request 400s:
  `result: "API Error: 400 thinking.enabled.budget_tokens: Input should be greater than or equal to 1024"`,
  `is_error:true`, `output_tokens:0`, exit 1. Re-running at cap 2048 → exit 0, 441 tokens.
- **glm-5.1** is the true salvage case: a completed result (`output_tokens:256`, `terminal_reason:"completed"`)
  that the old `r.status !== 0` bail discarded before parsing.

**The change** (`bin/fl-bench.mjs`):
1. `extractClaudeResult(stdout)` — parses the claude `--output-format json` array/stream and returns
   `{ parsed, outputTokens, inputTokens, isError, resultText }`. `is_error`/`resultText` are read **only from
   the final `type:"result"` element** (not a whole-tree OR); input tokens sum `input_tokens +
   cache_read + cache_creation` (so the hot call's cached re-read still counts).
2. Pure `decideClaudeOutcome(status, ex, timeoutSecs)` — **salvages a completed result (`outputTokens>0`)
   regardless of CLI exit code, checked BEFORE the `is_error` branch**; surfaces the real API-error reason
   when there are no tokens; keeps timeout (124) and unparseable/empty failures.
3. Raised the soft output cap to **2048 (light) / 8192 (heavy)** — both clear the 1024 thinking floor.
4. Captured input tokens for every provider (claude usage incl. cache; local `prompt_tokens`; codex
   `token_count`) and added `profile`, `cold_input_tokens`, `hot_input_tokens` to each JSONL row; the
   table now shows `cIn`/`cOut`/`hOut`.

**Heavy workload (the second half of the request).** New `--profile light|heavy` (default light;
`--heavy`/`--light` shorthand). `heavy` injects a deterministic **>5k-token** code context
(`buildHeavyContext`, 30 handlers) plus a long-output instruction (cap 8192, timeouts 600/1200) so the
benchmark measures a realistic coding/agentic load instead of light's few-hundred-token decode.

**Adversarial-review catch (fixed before commit).** The first cut set `isError` via a whole-tree OR and
checked it *before* salvage, so a completed run with an intermediate `is_error:true` stream event (or an
`error_max_turns` result that still produced tokens) was mis-scored FAIL — **re-introducing D-0005**. Fixed
by (1) reading `is_error` only from the final result element and (2) ordering salvage before the error
branch. Both paths are regression-tested.

**Verification.**
- `bin/fl-bench-salvage.test.mjs` — **15/15**, importing the REAL exported helpers (`extractClaudeResult`,
  `decideClaudeOutcome`, `buildHeavyContext`, `PROFILES`). Covers: glm-5.1 salvage on nonzero exit, haiku
  `is_error` reason, the two review-regression scenarios (intermediate error + completed final; `is_error`
  result WITH tokens), cached-input summation, line-by-line fallback, timeout, empty, the cap≥1024 invariant,
  and the heavy >5k input/output targets.
- **Live, light:** `haiku` now passes — cold 60.3 / hot 59.6 t/s (was the headline FAIL).
- **Live, heavy** (`--profile heavy`, cold+hot): input AND decode both clear 5k —
  `local Qwen3-Coder-30B` **cIn 9,918 / out 8,192** (was "a few hundred" under light),
  `minimax-m3` **cIn 74,105 / cOut 15,246 / hOut 13,298**, `opus` **cIn 35,943 / hOut 11,668** (cold 3,128 — model chose a concise complete answer; the cap allows more).

Follow-up still open (NOT done here): the tournament runners (`glm-run.sh` / `minimax-run.sh` / codex /
