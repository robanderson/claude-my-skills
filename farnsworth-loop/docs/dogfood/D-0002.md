# D-0002 — Round-1 `verdict`/`guidance` silently not written on large two-pass runs (P2 persist no-op)

- id: D-0002
- status: open
- severity: sev2            # outcomes still recoverable (mapping.json + final verdict persist), but the round-1 audit trail is lost
- area: infra
- discovered: 2026-06-15
- reporter: rob
- provenance (NON-DURABLE, may be gc'd): run-id `fl-dogfood-system-20260615-115049` (two-pass, N=8,
  the run that designed THIS backlog). Reproduced live; see the directory listing in the excerpt.

## Problem

The verdict-persistence feature (PR #2) writes files by handing a single large shell command to a
cheap **haiku agent** (`persist()` in `workflows/tournament.mjs`), because the workflow sandbox has
no `node:fs` (see [D-0001](D-0001.md)). On a real large two-pass run, the **P2 round-1 write**
(`review-1/verdict.json`, `review-1/verdict.md`, `review-1/guidance.md`) **was silently skipped** —
the haiku write-agent returned success but did not actually execute its (large) command. The
later P6 write (`review-final/verdict.*`, `SUMMARY*.md`, `mapping.json`) succeeded.

Crucially `persist()` logged **no** failure (its `try/catch` saw the agent "succeed"), and there
were **no** `.partial` leftovers — so the loss is invisible without inspecting the run dir. The
smoke tests in PR #2 passed only because their content was tiny (a 3-line haiku); the failure is
correlated with payload size and/or plain LLM non-determinism. Routing file writes through an
unverified LLM agent is the root cause.

## Durable evidence (verbatim excerpt — survives `.runs/` deletion)

> `ls review-1/` for run `fl-dogfood-system-20260615-115049`:
> `_pool.md  A/ B/ C/ D/ E/ F/ G/ H/`   ← **no `verdict.json`, `verdict.md`, or `guidance.md`**
>
> Same run, `review-final/` and top level: `verdict.json  verdict.md  _pool.md  …` present;
> `mapping.json`, `SUMMARY.md`, `SUMMARY.blind.md` present. `find . -name '*.partial'` → none.
> Workflow logs contain **no** `persist failed (Review)` line.

So P6 persisted; P2 did not; and nothing surfaced the loss.

## Repro

1. Run a real `@@FL:8:2` (two-pass, large multi-candidate review with guidance — i.e. a big P2
   payload), not a toy task.
2. After it completes, `ls .runs/<run-id>/review-1/`.
3. Observe `verdict.json` / `verdict.md` / `guidance.md` are missing while `review-final/` +
   `mapping.json` + `SUMMARY*.md` are present, with no error logged.

## Suspected fix (for the worker who claims this)

`persist()` must **verify** the write instead of trusting the agent: have the persist agent run the
command **and** report back `wc -c` (or an `ls`) for each target path, then the workflow checks each
file exists with non-zero size and **retries once** on a miss (logging a real failure if it still
misses). Cheaper alternatives to consider: split each file into its own `persist()` call (smaller
commands), or chunk very large content. Whatever the mechanism, an unverified LLM write must never
be treated as success.
