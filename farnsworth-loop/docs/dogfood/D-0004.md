# D-0004 — Non-Anthropic round-1 winner is wrongly excluded from the final pool (carryover provenance re-check)

- id: D-0004
- status: open
- severity: sev1            # changes who wins: a valid round-1 winner is dropped from the final ranking, biasing finals toward Anthropic / round-2 candidates
- area: review
- discovered: 2026-06-15
- reporter: rob
- provenance (NON-DURABLE, may be gc'd): run-id `fl-bench-20260615-115528` (workflow `wf_58ecb8f5-303`),
  two-pass N=8 (2 opus / 2 sonnet / 2 codex-high / 2 minimax) — the run that built the `fl-bench` feature.

## Problem

In `workflows/tournament.mjs`, the two-pass carried-over round-1 winner is re-validated for the final
pool against a workspace whose provenance log has **already been stripped**, so any **non-Anthropic**
round-1 winner is marked invalid and silently excluded from the final ranker.

Path through the code:

1. Round 1: each attempt's `c.ws` = `round-1/candidate-<i>/` (still contains its `_codex_run.log` /
   `_glm_run.log` / `_minimax_run.log` / `_local_run.log`).
2. `stageAndValidate(...)` copies each into `review-1/<blind>/` and **deletes the engine logs by name**
   (the staging allowlist), then returns each candidate with `ws` **re-pointed to the stripped**
   `review-1/<blind>/` dir.
3. The champion is `blind1.find(c => c.blind === review.winner)`, so `champ.ws` is that **stripped**
   `review-1/<blind>/` dir, and `champ.dispatch` is preserved (e.g. `codex`).
4. The final pool carries `{ ws: champ.ws, dispatch: champ.dispatch, round: 1 }`. `stageAndValidate`
   runs again for `review-final/`, and for a runner-backed `dispatch` it requires the provenance
   contract in `${c.ws}/_<prov>_run.log` — **but that log was deleted in step 2** → `P=0` → `valid:false`.

Native **Anthropic** carryovers are unaffected: their `dispatch` has no provenance log, so the check is
`P=1` unconditionally — which is exactly why the bug is invisible until a GLM/codex/MiniMax/local
attempt wins round 1. Result: a legitimately-winning non-Anthropic candidate never reaches the final
Opus ranker, and the final winner is chosen from a pool it was unfairly removed from.

## Durable evidence (verbatim excerpt — survives `.runs/` deletion)

Round 1 — the winner was a **non-Anthropic** candidate (codex-high), and the blind reviewer called it
the only end-to-end-verified entry:

> `round1.review.winner = "E"` · `round1.ranking = ["E","A","H","G","C","F","B"]`
> `mapping.json` (round 1): `{"candidate":"E","model":"codex-high","valid":true}`
> verdict (Candidate E, round 1): *"VERIFIED WORKING END-TO-END on the real system: live --list-models
> discovered the actual 20-model omlx catalogue, and a live `--models haiku` run produced real provider
> token counts (cold 474 / hot 672), correct cold/hot/delta, and a complete JSONL record."*

Final pool — that same winner, carried over as round-1, was dropped on a provenance check:

> `mapping.json` (final): `{"candidate":"G","model":"codex-high","round":1,"valid":false,"failReason":"provenance check failed (timeout/error/empty)"}`
> `final.rank.winner = "H"` · `final.winnerRound = 2` (an opus round-2 attempt)
> `final.ranking = ["H","I","F","A","C","B","E","D"]`   ← the carried-over round-1 winner (blind `G`) is **absent** (excluded as invalid)

The carryover produced a real, validated deliverable in round 1; its only "failure" is that the
re-validation looked for a log the engine itself had already removed.

## Repro

1. Run a two-pass `@@FL` whose **round-1 winner is a non-Anthropic model** (GLM / codex / MiniMax /
   local) — e.g. a mixed field where a codex/minimax attempt wins round 1.
2. Inspect the final mapping: the carried-over round-1 winner row (`round:1`) shows
   `valid:false, failReason:"provenance check failed (timeout/error/empty)"`.
3. Confirm it is missing from `final.ranking` — it never reached the ranker.
   (Contrast: when an Anthropic attempt wins round 1, the carryover is `valid:true` and competes — so
   the bug only manifests for non-Anthropic round-1 winners.)

## Suspected fix (for the worker who claims this)

The carryover was **already validated in round 1**; re-running the provider provenance check against a
deliberately-stripped staging dir is the defect. Two clean options:

- **Trust the prior validation (preferred, smallest):** mark the carried-over finalist as
  provenance-valid by construction — e.g. tag the final-pool carryover entry (`carriedOver:true`) and
  have `stageAndValidate` skip the provenance grep for it (still require a non-empty deliverable, which
  the staged dir has). It passed provenance once; it cannot "un-pass."
- **Stage from the original workspace:** carry the champion's **original** `round-1/candidate-<i>/`
  path (which still has its `_<prov>_run.log`) into the final pool instead of the stripped
  `review-1/<blind>/` dir, so re-validation has the log it needs. Requires preserving the pre-staging
  `ws` (e.g. an `origWs` field on the staged record).

Either way, add a regression case: a two-pass run whose round-1 winner is runner-backed must appear,
`valid:true`, in `review-final`'s pool and `final.ranking`.
